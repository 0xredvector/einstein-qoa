## Anúncio Oficial

A Einstein anuncia oficialmente a adoção do OpenTelemetry (OTel) como padrão corporativo de observabilidade. A partir de 1º de março de 2026, todos os novos serviços devem utilizar bibliotecas OpenTelemetry para instrumentação, e os serviços existentes terão 6 meses para conformidade.

## O Que é OpenTelemetry?

OpenTelemetry é um projeto de código aberto gerenciado pela Cloud Native Computing Foundation (CNCF) que fornece APIs, SDKs e ferramentas padronizadas para coleta de telemetria. Oferece:

- **APIs Agnósticas**: Mesmo código funciona independente do backend (Jaeger, Datadog, New Relic, etc.)
- **Instrumentação Automática**: Bibliotecas nativas para populares frameworks (Spring, FastAPI, Express)
- **Padrão Aberto**: Não há vendor lock-in, garantindo flexibilidade futura
- **Comunidade Ativa**: Milhares de contribuidores, amplo ecossistema de integrações

## Por Que Adotamos OpenTelemetry?

Durante os últimos 12 meses, o time de arquitetura conduziu análise comparativa entre OpenTelemetry, alternativas proprietárias e soluções customizadas. Os resultados foram conclusivos:

### Padronização
Antes, diferentes teams utilizavam abordagens distintas de observabilidade: alguns com bibliotecas proprietárias, outros com instrumentação manual customizada. Isso criava:
- Curva de aprendizado elevada para novos engenheiros
- Inconsistência em formato de dados entre serviços
- Dificuldade em correlação de traces entre sistemas

OpenTelemetry elimina essa fragmentação com padrão único.

### Flexibilidade Operacional
Nossas ferramentas de observabilidade evoluem: mudamos de Datadog para stack interno Prometheus + Grafana + Jaeger. Com soluções proprietárias, seríamos presos a apenas um backend.

Com OpenTelemetry, podemos:
- Trocar backends sem reescrever instrumentação
- Enviar mesmos dados simultaneamente para múltiplos destinos
- Adotar novas ferramentas conforme surgem inovações

### Custo
Eliminamos dependência de SDKs proprietários caros. OpenTelemetry é software livre com custo apenas de infraestrutura para rodar collectors e backends.

### Tamanho da Comunidade
OpenTelemetry é padrão de facto em indústria. Empresas como Google, Microsoft, Uber e Netflix já utilizam. Isso significa:
- Mais integrações disponíveis
- Mais bibliotecas de instrumentação automática
- Mais recursos educacionais
- Maior demanda por skill no mercado

## O Que Muda para Desenvolvedores?

### Antes (Sem Padrão)

```java
// Cada serviço tinha sua abordagem
logger.info("User login", userId, timestamp);
// Sem correlação entre serviços
// Sem traces distribuídos automáticos
```

### Depois (Com OpenTelemetry)

```java
@Configuration
public class OpenTelemetryConfig {
    @Bean
    public OpenTelemetry openTelemetry() {
        return OpenTelemetryBuilder.create()
            .addSpanProcessor(
                BatchSpanProcessor.builder(OtlpGrpcSpanExporter.builder()
                    .setEndpoint("http://otel-collector:4317")
                    .build())
                .build())
            .build();
    }
}

// Instrumentação automática através de annotations
@WithSpan("processUserLogin")
public void loginUser(String userId) {
    Span span = Span.current();
    span.setAttribute("user.id", userId);
    // Traces e logs são automaticamente correlacionados
}
```

## Benefícios Imediatos

### Melhor Observabilidade
- Traces distribuídos correlacionam chamadas entre 10+ serviços
- Root cause analysis de incidentes é 5x mais rápida
- Latência por camada (banco, cache, API) é automaticamente medida

### Debugging Mais Fácil
- Logs são automaticamente correlacionados com traces
- Context baggage passa automaticamente entre serviços
- Identificar serviço culpado por lentidão é trivial

### Conformidade com Padrões Internacionais
- OpenTelemetry é recomendado por OWASP para observabilidade
- Alinha com requisitos de compliance e auditoria
- Facilita certificações como ISO 27001

## Perguntas Frequentes

**P: Meu serviço já tem instrumentação customizada. Preciso reescrever?**
R: Não imediatamente. Para serviços estáveis, continuarão funcionando normalmente até Agosto. Para novos recursos, use OpenTelemetry desde já.

**P: OpenTelemetry adiciona overhead de performance?**
R: Overhead é negligenciável (~1% CPU) com sampling apropriado. Recomendamos sample 100% em desenvolvimento, 10% em produção.

**P: E se eu usar linguagem X que não tem SDK OpenTelemetry?**
R: Entre em contato conosco. OpenTelemetry suporta 12+ linguagens. Se sua linguagem não está suportada, avaliaremos alternativas.

**P: Isso muda onde envio meus logs?**
R: Não. Logs continuam em Loki com mesma estrutura. OpenTelemetry apenas adiciona correlação com traces.
